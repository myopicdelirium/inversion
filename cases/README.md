# The case register

Retrodiction targets, prepared before the capability to run them exists. The point of this directory is its timestamps: every dossier committed here has a git history older than the engine that will eventually be tested against it, so no pass/fail bar can be accused of chasing a result. Dossiers are mirrored on the site's ledger with register numbers and, on freezing, Zenodo DOIs: three independent timestamp anchors, one of them inside the same audit chain as the engine and its constitution.

Discipline, binding once a dossier is frozen:

1. Every dossier follows the fixed schema (kept with the dossiers; drafted by the program's strategy track). Seven sections: signature, pressure schedule, contrast cases, preregistered pass/fail measures with their statistical form, capability checklist, policy levers, and the observables dictionary mapping model quantities to historical ones.
2. **The firewall.** Each dossier's quantitative content is split into parameterization evidence (usable for fitting) and outcome evidence (the signature, untouchable until judgment). Runs against a dossier must be produced by the compiler protocol (docs/COMPILER-PROTOCOL.md), committed with seeds and config hashes before the signature comparison is unblinded. The timestamp defends the criterion; the firewall defends the fitting.
3. Pass/fail bars specify pooling: seed counts, statistics, and what counts as replication. An ambiguous bar is a bug.
4. Failures are published with the same prominence as passes, per the constitution's standing practice.

Ladder: tier one, bounded communities (Jonestown, Heaven's Gate, the Xhosa cattle-killing); tier two, city scale (Munster 1534); tier three, macro-societies (Weimar and beyond). Fiction targets (1984) are a separate expressiveness register, never sold as empirical.
